# 对话站单点登录集成开发指南

本文档面向希望接入对话站的第三方平台开发者，说明如何实现单点登录（SSO），以及对接过程中需要传递哪些参数。

这里的“第三方平台”指身份源平台，例如 LinkCode、业务后台、会员系统、API 中转站等；“对话站”指基于 LobeHub 或类似架构的聊天应用。

## 一、对接目标

集成完成后，应实现以下体验：

- 用户已登录第三方平台时，点击“对话站”入口即可自动登录对话站。
- 用户未登录第三方平台时，也可以直接打开对话站。
- 用户在对话站开始对话或访问需要登录的能力时，对话站提示登录，并跳转到第三方平台登录。
- 第三方平台登录完成后，自动回到对话站。
- 如有需要，第三方平台可以在 SSO 过程中把用户可用的模型 API Key 一并下发给对话站。

## 二、整体流程

推荐使用“短时一次性 code + 服务端换码”的方式。

1. 用户在第三方平台点击“对话站”。
2. 第三方平台后端确认用户已登录。
3. 第三方平台创建一个短时一次性 `code`。
4. 浏览器跳转到对话站 callback 地址，并携带 `code`。
5. 对话站后端拿 `code` 请求第三方平台的 exchange 接口。
6. 第三方平台校验共享密钥和 `code`，返回用户信息和可选 API Key。
7. 对话站根据邮箱创建或登录本地用户，并建立自己的 session。
8. 对话站跳转回用户原本要访问的页面。

示例跳转地址：

```text
https://chat.example.com/linkcode/sso/callback?code=一次性code&returnTo=/
```

> `linkcode` 只是当前实现中的 callback path 名称。第三方平台可以和对话站约定自己的 callback path。

## 三、角色分工

| 角色 | 负责事项 |
|---|---|
| 第三方平台 | 用户登录、创建一次性 code、提供 exchange 接口、返回用户信息和可选 API Key。 |
| 对话站 | 接收 callback、服务端换码、创建本地 session、保存 provider key、控制未登录提示。 |
| 浏览器 | 只负责页面跳转，不直接调用 exchange 接口。 |

## 四、第三方平台需要提供的接口

### 1. authorize 接口

该接口由第三方平台自己的前端或后端调用，用来创建一次性 SSO code，并返回对话站 callback URL。

```http
POST /api/v1/chat-sso/authorize
Authorization: Bearer <用户登录态>
Content-Type: application/json
```

请求参数：

```json
{
  "return_to": "/"
}
```

响应示例：

```json
{
  "redirect_url": "https://chat.example.com/linkcode/sso/callback?code=abc123&returnTo=/",
  "expires_at": "2026-06-20T12:00:00Z"
}
```

参数说明：

| 参数 | 位置 | 是否必填 | 说明 |
|---|---|---:|---|
| `return_to` | request body | 否 | 用户登录对话站后要回到的对话站路径。建议只允许相对路径。 |
| `redirect_url` | response body | 是 | 第三方平台生成的对话站 callback URL，必须包含一次性 `code`。 |
| `expires_at` | response body | 建议 | code 过期时间，便于前端或日志排查。 |

实现要求：

- 只有已登录用户才能调用 authorize。
- `code` 必须随机、不可预测。
- `code` 必须短时有效，建议 120 秒。
- `code` 必须一次性消费。
- `code` 建议存 Redis，并设置 TTL。

### 2. exchange 接口

该接口只允许对话站后端调用，用一次性 `code` 换取用户信息和可选 API Key。

```http
POST /api/v1/chat-sso/exchange
X-Chat-SSO-Secret: <共享密钥>
Content-Type: application/json
```

请求参数：

```json
{
  "code": "abc123"
}
```

响应示例：

```json
{
  "user": {
    "id": "123",
    "email": "user@example.com",
    "username": "user",
    "avatar_url": "https://example.com/avatar.png"
  },
  "api_base_url": "https://api.example.com",
  "keys": [
    {
      "provider": "gpt",
      "platform": "openai",
      "key": "sk-xxx",
      "group_id": "openai-chat"
    },
    {
      "provider": "claude",
      "platform": "anthropic",
      "key": "sk-xxx",
      "group_id": "anthropic-chat"
    },
    {
      "provider": "gemini",
      "platform": "gemini",
      "key": "sk-xxx",
      "group_id": "gemini-chat"
    }
  ]
}
```

参数说明：

| 参数 | 位置 | 是否必填 | 说明 |
|---|---|---:|---|
| `X-Chat-SSO-Secret` | request header | 是 | 第三方平台和对话站约定的共享密钥。必须只在服务端使用。 |
| `code` | request body | 是 | authorize 阶段生成的一次性 code。 |
| `user.email` | response body | 是 | 对话站用于创建或匹配用户的核心字段。 |
| `user.id` | response body | 建议 | 第三方平台用户 ID，可用于审计或绑定关系。 |
| `user.username` | response body | 否 | 用户显示名。 |
| `user.avatar_url` | response body | 否 | 用户头像 URL。 |
| `api_base_url` | response body | 有 key 时建议 | 第三方平台或中转站的 API 网关根地址。 |
| `keys` | response body | 否 | 要同步给对话站的 provider key 列表。可以为空数组。 |
| `keys[].provider` | response body | 有 key 时必填 | 对话站 provider 标识，例如 `gpt`、`claude`、`gemini`。 |
| `keys[].platform` | response body | 建议 | 第三方平台内部平台标识，例如 `openai`、`anthropic`、`gemini`。 |
| `keys[].key` | response body | 有 key 时必填 | 对话站要保存的 API Key。 |
| `keys[].group_id` | response body | 否 | 第三方平台侧分组 ID 或分组名，便于排查。 |

实现要求：

- exchange 必须校验共享密钥。
- exchange 必须服务端调用，不要允许浏览器直接调用。
- exchange 成功后必须立刻消费 `code`，重复请求应失败。
- 如果 `code` 不存在、过期或已使用，应返回未授权或 code 无效错误。
- 返回的 `user.email` 应稳定且唯一。

## 五、对话站需要实现什么

对话站侧通常需要实现：

1. callback route：接收 `code` 和 `returnTo`。
2. exchange client：后端调用第三方平台 exchange 接口。
3. user upsert：根据 `user.email` 创建或匹配本地用户。
4. session 创建：创建对话站自己的登录态。
5. provider key 同步：把 `keys` 写入对应 provider 的配置。
6. 登录入口替换：对话站自己的登录/注册入口应跳到第三方平台。
7. 未登录触发登录：用户开始对话时，如果未登录，应提示并跳转第三方平台登录。

## 六、最小可用返回

如果第一阶段只想打通登录，可以不返回 API Key。

exchange 最小响应：

```json
{
  "user": {
    "id": "123",
    "email": "user@example.com",
    "username": "user"
  },
  "keys": []
}
```

后续再扩展 `api_base_url` 和 `keys` 即可。

## 七、推荐配置项

第三方平台建议配置：

```yaml
chat_sso:
  enabled: true
  shared_secret: "至少32字节强随机密钥"
  chat_base_url: "https://chat.example.com"
  callback_path: "/linkcode/sso/callback"
  code_ttl_seconds: 120
  api_base_url: "https://api.example.com"
  auto_create_api_keys: true
```

对话站建议配置：

```env
SSO_ENABLED=1
SSO_BASE_URL=https://platform.example.com
SSO_LOGIN_URL=https://platform.example.com/login
SSO_EXCHANGE_URL=https://platform.example.com/api/v1/chat-sso/exchange
SSO_SHARED_SECRET=双方一致的强随机密钥
```

## 八、returnTo 安全规则

`returnTo` 用于控制登录后回到哪里。必须做安全限制：

- 推荐只接受相对路径，例如 `/`、`/chat`、`/settings`。
- 如果允许绝对 URL，必须限制为对话站同源 URL。
- 禁止跳转到任意第三方域名，防止开放跳转漏洞。
- 空值或非法值应回退到 `/`。

## 九、API Key 同步建议

如果第三方平台要把模型 key 同步给对话站，建议使用稳定 provider 名称：

| provider | 说明 |
|---|---|
| `gpt` | OpenAI/GPT provider。 |
| `claude` | Anthropic/Claude provider。 |
| `gemini` | Google Gemini provider。 |

`api_base_url` 建议填写网关根地址，不要直接填写具体接口路径。

示例：

```text
https://api.example.com
```

对话站可以按 provider 自行决定是否追加 `/v1`、`/v1beta` 等路径。

## 十、安全要求

- 共享密钥必须只存在服务端，不允许暴露到前端。
- SSO code 必须短时有效，建议 30 到 600 秒。
- SSO code 必须一次性使用。
- exchange 接口建议只允许对话站服务器 IP 或可信网络访问。
- 所有生产环境请求必须使用 HTTPS。
- 日志中不要记录完整 `code`、API Key 或共享密钥。
- API Key 返回给对话站后，对话站应加密保存。

## 十一、错误码建议

| 错误 | 场景 |
|---|---|
| `SSO_DISABLED` | SSO 未启用。 |
| `SSO_UNAUTHORIZED` | 共享密钥缺失或错误。 |
| `SSO_CODE_INVALID` | code 不存在、过期或已使用。 |
| `SSO_USER_INACTIVE` | 用户不存在或已被禁用。 |
| `SSO_EXCHANGE_FAILED` | 对话站调用 exchange 失败。 |

## 十二、开发联调清单

1. 第三方平台已登录用户调用 authorize，能拿到 `redirect_url`。
2. 浏览器访问 `redirect_url`，对话站 callback 能收到 `code`。
3. 对话站后端调用 exchange，第三方平台能返回 `user.email`。
4. 对话站能创建 session，并在页面上显示已登录状态。
5. 未登录用户在对话站开始对话时，会跳转第三方平台登录。
6. 登录完成后能回到对话站原页面。
7. 如果返回 `keys`，对话站能写入并启用对应 provider。
