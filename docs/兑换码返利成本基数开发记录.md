# 兑换码返利成本基数开发记录

## 需求

余额兑换码可同时记录用户到账面值和实际收款成本。例如用户兑换后到账 278，实际收款为 200；邀请返利应按 200 计算，而不是按 278 计算。

## 实现

- 新增 `redeem_codes.affiliate_rebate_base_amount`，类型为 `DECIMAL(20,8)`。
- `value` 始终表示用户实际到账余额；新字段只用于计算邀请返利。
- 新生成的正数余额兑换码未填写成本时，后端会保存与 `value` 相同的返利基数，保持原有返利规则。
- 成本仅允许设置在 `0` 到面值之间；`0` 表示赠送余额且不产生邀请返利。
- 并发、订阅、邀请码和负数余额退款码不能设置返利成本。
- 历史兑换码的字段为空时，兑换逻辑自动回退按 `value` 返利，因此无需修改已兑换兑换码，也不会改变已经发放的返利。
- 管理员兑换码生成页面新增“返利计算金额（实收/成本）”输入框和“返利基数”列表列；普通用户接口不返回该内部字段。

## 发布顺序

1. 部署本次代码。`185_redeem_code_affiliate_rebate_base.sql` 会自动创建新字段。
2. 确认服务启动完成后，再按下方 SQL 修改历史未兑换的促销兑换码。
3. 不要更新 `status = 'used'` 的兑换码，已产生的返利记录不应被追溯修改。

## 历史兑换码 SQL

先将所有旧的、未兑换的正数余额兑换码显式设为按面值返利。此步骤不是功能运行的前置条件，但能让历史码与新生成的默认兑换码保存相同的语义。

```sql
BEGIN;

UPDATE redeem_codes
SET affiliate_rebate_base_amount = value
WHERE type = 'balance'
  AND status = 'unused'
  AND used_by IS NULL
  AND used_at IS NULL
  AND value > 0
  AND affiliate_rebate_base_amount IS NULL;

COMMIT;
```

再按兑换码或 ID 精确把促销码改为实际成本。以下示例中，到账 278 的兑换码按 200 计算邀请返利：

```sql
BEGIN;

UPDATE redeem_codes
SET affiliate_rebate_base_amount = 200.00000000
WHERE type = 'balance'
  AND status = 'unused'
  AND used_by IS NULL
  AND used_at IS NULL
  AND code IN ('兑换码1', '兑换码2');

COMMIT;
```

执行前可先核对目标数据：

```sql
SELECT id, code, value, affiliate_rebate_base_amount, status, used_by, used_at
FROM redeem_codes
WHERE code IN ('兑换码1', '兑换码2')
ORDER BY id;
```
