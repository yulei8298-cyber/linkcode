# 兑换码返利成本基数开发记录

## 需求

余额兑换码可同时记录用户到账面值和实际收款成本。例如用户兑换后到账 278，实际收款为 200；邀请返利应按 200 计算，而不是按 278 计算。

## 实现

- 新增 `redeem_codes.affiliate_rebate_base_amount`，类型为 `DECIMAL(20,8)`。
- `value` 始终表示用户实际到账余额；新字段只用于计算邀请返利。
- 新生成的正数余额兑换码未填写成本时，后端会保存与 `value` 相同的返利基数，保持原有返利规则。
- 成本仅允许设置在 `0` 到面值之间；`0` 表示赠送余额且不产生邀请返利。
- 并发、订阅、邀请码和负数余额退款码不能设置返利成本。
- 历史兑换码的字段为空时，兑换逻辑自动回退按 `value` 返利，因此无需修改已兑换兑换码，也不会改变已经发放的返利。
- 管理员兑换码生成页面新增“返利计算金额（实收/成本）”输入框和“返利基数”列表列；普通用户接口不返回该内部字段。

## 历史兑换码回填

部署时会自动依次执行 `185_redeem_code_affiliate_rebate_base.sql` 和 `186_backfill_historical_redeem_code_rebate_bases.sql`。其中 `186` 已按确认的促销档位回填历史未兑换余额码的返利成本：

| 兑换码到账面值 | 返利计算金额（实收/成本） |
| --- | --- |
| 20 | 20 |
| 68 | 50 |
| 138 | 100 |
| 278 | 200 |

回填条件固定为：余额类型、`status = 'unused'`、`used_by` 与 `used_at` 均为空，并且此前未设置返利基数。因而它不会改动已兑换码及其已经产生的返利，也不会覆盖管理员后来为兑换码手动设置的成本。

不在上述四档内的历史余额兑换码保留空返利基数，兑换时继续按到账面值 `value` 计算返利，保持旧行为。若后续确认其他促销档位，需要单独新增迁移或按兑换码精确更新，不能批量猜测成本。
