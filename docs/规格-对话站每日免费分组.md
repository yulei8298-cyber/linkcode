## 目标

**问题：** 当前对话站依赖 `OpenAI-chat` 免费订阅和预创建密钥，用户仍能看到并误把该分组密钥用于 Codex；分组限制又依赖名称，免费额度无法按余额用户每日独立核算。

**方案：** 给分组增加隐藏、免费、每日免费额度和仅对话站四项配置；中转站用共享密钥识别对话站，在首次无 API Key 请求时按平台自动选组、创建隐藏密钥并完成请求，Lobe 保存响应返回的密钥。免费用量按用户、分组、自然日记账，不扣余额，也不依赖订阅。

## 边界与约束

**始终遵守：** 保留两端现有二开；运行时不用分组名判断权限；用户 ID 仅在共享密钥校验后可信；免费组限标准模式且额度大于零；隐藏组对用户不可选、不可绑定、不可见，管理员仍可管理；实际成本照常记录；Lobe 按本次触碰文件打包，不依赖 Git；开发验收阶段不提交不推送，收到用户明确指令后再提交。

**需先确认：** 若实现中发现无法识别旧 `OpenAI-chat` 自动订阅而必须删除其他分组或人工付费订阅，停止并确认；若现有模型运行时不支持安全拦截响应头，需要改变对话站与中转站的响应协议，停止并确认。

**绝不执行：** 不降低普通密钥鉴权；不让客户端自报来源；不把共享密钥或自动密钥下发浏览器；不复用订阅额度表；不覆盖 Lobe 无关改动；不改变普通组计费。

## 输入输出与边界场景

| 场景 | 输入 / 状态 | 预期行为 | 错误处理 |
|---|---|---|---|
| 隐藏组管理 | 管理员开启“用户隐藏” | 管理端可见，用户可用分组和密钥列表不可见 | 用户创建或改绑返回禁止绑定 |
| 免费请求 | 标准组开启免费且当天有额度 | 记录成本和免费日用量，不扣余额 | 不触发余额不足通知 |
| 额度耗尽 | 当日累计已达到额度 | 请求在调度前拒绝 | HTTP 429，`当前每日免费额度已用完，请明天再使用` |
| 来源限制 | `chat_station_only=true` | 仅共享密钥匹配时放行 | HTTP 403，中文提示只能从对话站使用 |
| 首次 OpenAI/Claude 请求 | 无密钥，带共享密钥和用户 ID | 按平台选组，复用或创建隐藏密钥并继续 | 响应头供 Lobe 服务端保存密钥 |
| 无候选组 | 无匹配的活跃免费对话站组 | 不创建密钥、不调度 | HTTP 401，Lobe 显示前往设置密钥 |
| 已有密钥失效 | 密钥所属分组已删除或不存在 | 视为密钥无效 | HTTP 401，中文密钥错误 |
| 旧免费订阅 | 旧 `OpenAI-chat` 自动订阅 | 停止创建并清理该分组订阅 | 不影响其他订阅 |

**验收标准：**

- Given Lobe Provider 无密钥，when 首次发送 OpenAI/Claude 消息且有候选组，then 同次获得回答并加密保存返回密钥。
- Given 普通余额组或订阅组，when 调用，then 原鉴权与计费不变。

## 设计说明

账本累计最终 `actual_cost`；请求前检查，达到额度即阻断。并发可轻微越界，但后续请求必须拒绝。候选组须满足 active、standard、is_free、chat_station_only、平台和额度有效，按 `sort_order,id` 选择。
